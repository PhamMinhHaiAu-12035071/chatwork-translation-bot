# Repository Guidelines

## Project Structure & Module Organization

This repository is a Bun + TypeScript monorepo using workspaces under `packages/*`.

- `packages/core/src`: shared domain types, translation interfaces/services, and parsing utilities.
- `packages/bot/src`: runtime app code (HTTP server, Chatwork client, webhook router/handler, env validation).
- `dist/server.js`: build output generated by `bun run build` (do not edit manually).
- Root configs: `eslint.config.ts`, `.prettierrc`, `commitlint.config.ts`, `docker-compose.yml`.

Keep business logic in `core` and integration/transport concerns in `bot`.

## Build, Test, and Development Commands

- `bun install`: install workspace dependencies.
- `bun run dev`: run the bot from `packages/bot/src/index.ts` (development mode).
- `bun run build`: bundle bot to `dist/server.js` for deployment.
- `bun run typecheck`: run strict TypeScript checks for root, core, and bot configs.
- `bun run lint` / `bun run lint:fix`: run ESLint, optionally auto-fix.
- `bun run format`: format TypeScript files with Prettier.
- `bun test`: run all tests with Bun test runner.
- `docker compose up --build`: run the service locally in Docker on port `3000`.

## Coding Style & Naming Conventions

- Language: TypeScript (ESM), strict mode enabled.
- Formatting: 2 spaces, single quotes, no semicolons, trailing commas (`.prettierrc`).
- Lint rules enforce `type` imports and disallow unused vars unless prefixed with `_`.
- Naming conventions:
  - `PascalCase` for types/classes.
  - `camelCase` for variables/functions.
  - `kebab-case.ts` for file names (example: `parse-command.ts`).

## Testing Guidelines

- Framework: Bun test (`import { describe, it, expect } from 'bun:test'`).
- Test files should be colocated and named `*.test.ts` (example: `parse-command.test.ts`).
- Prioritize coverage for parsing, webhook validation, and error paths.
- Before pushing, run: `bun test && bun run typecheck && bun run lint`.

## Commit & Pull Request Guidelines

No commit history exists yet; follow enforced Conventional Commits via commitlint.

- Allowed commit types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `perf`, `ci`, `build`, `revert`.
- Example: `feat(bot): handle webhook signature failures`.
- PRs should include:
  - Clear problem statement and change summary.
  - Validation evidence (commands run + outcomes).
  - Linked issue/task.
  - Notes for `.env` or API behavior changes (include sample payloads when relevant).

## Security & Configuration Tips

- Copy `.env.example` to `.env` and keep real tokens/secrets out of git.
- Required variables: `CHATWORK_API_TOKEN`, `CHATWORK_WEBHOOK_SECRET`, plus `PORT`/`NODE_ENV`.
- Verify runtime endpoints locally: `GET /health`, `POST /webhook`.
